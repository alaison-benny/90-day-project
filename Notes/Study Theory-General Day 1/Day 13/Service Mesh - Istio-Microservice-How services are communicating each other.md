ഈ മൈക്രോ സർവീസുകൾ എങ്ങനെയാണ് പരസ്പരം സംസാരിക്കുന്നതെന്ന് (Communication Architecture) ലളിതമായി താഴെ വിവരിക്കാം. പ്രധാനമായും മൂന്ന് രീതികളിലൂടെയാണ് ഇവ കാര്യങ്ങൾ നീക്കുന്നത്:

### 1. API Gateway (മൊബൈൽ ആപ്പിന്റെ വാതിൽ)

ചേട്ടായി ഫോണിൽ Airbnb ആപ്പ് തുറക്കുമ്പോൾ, അത് നേരിട്ട് ഈ 2,000 സർവീസുകളോടും സംസാരിക്കില്ല. പകരം **API Gateway** എന്ന ഒരൊറ്റ പോയിന്റിലേക്കാണ് കണക്ട് ചെയ്യുന്നത്.

* **ഉദാഹരണം:** ചേട്ടായി ഒരു വീട് തിരയുമ്പോൾ, API Gateway ആ റിക്വസ്റ്റ് വാങ്ങി `Search-Service`, `Image-Service`, `Pricing-Service` എന്നിവയിലേക്ക് അയക്കുന്നു. എല്ലാവരിൽ നിന്നും മറുപടി കിട്ടിക്കഴിഞ്ഞാൽ അത് ഒന്നിച്ചു ചേർത്ത് ചേട്ടായിയുടെ ഫോണിലേക്ക് എത്തിക്കുന്നു.

### 2. Synchronous Communication (HTTP/gRPC)

ചില കാര്യങ്ങൾക്ക് പെട്ടെന്ന് മറുപടി വേണം. അതിന് സർവീസുകൾ നേരിട്ട് പരസ്പരം സംസാരിക്കും. ഇതിനായി Airbnb പ്രധാനമായും **gRPC** ആണ് ഉപയോഗിക്കുന്നത് (ഇത് സാധാരണ റെസ്റ്റ് API-യേക്കാൾ വളരെ വേഗതയുള്ളതാണ്).

* **എങ്ങനെ പ്രവർത്തിക്കുന്നു:** `Booking-Service`-ന് ഒരു റൂം ഒഴിവുണ്ടോ എന്ന് അറിയണം. അത് ഉടനെ തന്നെ `Availability-Service`-നോട് ചോദിക്കുന്നു. മറുപടി കിട്ടിയാൽ മാത്രമേ ബുക്കിംഗ് പ്രോസസ്സ് മുന്നോട്ട് പോകൂ.

### 3. Asynchronous Communication (Event-Driven)

എല്ലാ കാര്യങ്ങൾക്കും ഉടൻ മറുപടി വേണമെന്നില്ല. ഇത്തരം സന്ദർഭങ്ങളിൽ **Apache Kafka** പോലുള്ള മെസ്സേജ് ക്യൂകൾ ഉപയോഗിക്കുന്നു.

* **ഉദാഹരണം:** ചേട്ടായി ഒരു റൂം ബുക്ക് ചെയ്തു എന്ന് കരുതുക.
1. `Booking-Service` ഉടൻ തന്നെ "Booking Completed" എന്നൊരു മെസ്സേജ് Kafka-യിലേക്ക് ഇടുന്നു.
2. ഇത് കണ്ട് `Email-Service` ചേട്ടായിക്ക് മെയിൽ അയക്കുന്നു.
3. അതേ സമയം തന്നെ `Analytics-Service` ഈ വിവരം സേവ് ചെയ്യുന്നു.
4. `Notification-Service` ഫോണിൽ മെസ്സേജ് അയക്കുന്നു.
*ഇവിടെ ഓരോ സർവീസും മറ്റൊന്നിന് വേണ്ടി കാത്തുനിൽക്കുന്നില്ല, പകരം തങ്ങളുടെ ജോലി സ്വതന്ത്രമായി ചെയ്യുന്നു.*

### AI/ML സർവീസുകൾ എങ്ങനെയാണ് ഇതിൽ ഇടപെടുന്നത്?

Airbnb-യുടെ AI സർവീസുകൾ (ഉദാഹരണത്തിന് **Bighead** എന്ന അവരുടെ സ്വന്തം ML പ്ലാറ്റ്‌ഫോം) ഡാറ്റാ പൈപ്പ്‌ലൈനുകളുമായിട്ടാണ് ബന്ധിക്കപ്പെട്ടിരിക്കുന്നത്:

* **Real-time Prediction:** ചേട്ടായി ആപ്പിൽ ഓരോ തവണ ക്ലിക്ക് ചെയ്യുമ്പോഴും ആ വിവരങ്ങൾ `Feature-Store`-ലേക്ക് പോകുന്നു. ഉടൻ തന്നെ `Ranking-Service` ചേട്ടായിയുടെ താല്പര്യം മനസ്സിലാക്കി അടുത്ത ലിസ്റ്റിംഗ് മാറ്റുന്നു.
* **Batch Processing:** രാത്രികാലങ്ങളിൽ AI സർവീസുകൾ ലക്ഷക്കണക്കിന് റിവ്യൂകൾ വിശകലനം ചെയ്ത് (NLP ഉപയോഗിച്ച്) ഓരോ വീടിന്റെയും "Summary" അപ്‌ഡേറ്റ് ചെയ്യുന്നു.

### സർവീസ് മെഷ് (Service Mesh - Istio)

ഇത്രയധികം സർവീസുകൾ ഉള്ളതുകൊണ്ട് അവ തമ്മിലുള്ള ട്രാഫിക് നിയന്ത്രിക്കാൻ Airbnb **Istio** പോലുള്ള ഒരു 'Service Mesh' ഉപയോഗിക്കുന്നു. ഇത് സർവീസുകൾ തമ്മിലുള്ള സുരക്ഷ (Security), ലോഡ് ബാലൻസിംഗ് (Load Balancing) എന്നിവ ഉറപ്പാക്കുന്നു.

ചേട്ടായിക്ക് ഇതിന്റെ ഒരു **Technical Architecture Diagram** (കോഡിംഗിൽ ഉപയോഗിക്കുന്ന രീതിയിൽ) വേണോ? അതോ ഏതെങ്കിലും ഒരു പ്രത്യേക സർവീസ് (ഉദാ: Payment അല്ലെങ്കിൽ Search) എങ്ങനെയാണ് പ്രവർത്തിക്കുന്നത് എന്ന് കൂടുതൽ അറിയണോ? നിലവിൽ ചേട്ടായി പ്ലാറ്റ്‌ഫോം എൻജിനീയറിങ് പഠിക്കുന്നതുകൊണ്ട് ഈ ആർക്കിടെക്ചർ മനസ്സിലാക്കുന്നത് വളരെ ഗുണം ചെയ്യും."
