## Teach me terraform ഫയലുകളുടെ നിയമങ്ങൾ (Indentation) in simple language and real world production examples

ഒരു സീനിയർ പ്ലാറ്റ്‌ഫോം എൻജിനീയർ എന്ന നിലയിൽ, ടെറാഫോം കോഡ് വൃത്തിയായി എഴുതുന്നത് (Indentation) വളരെ പ്രധാനമാണ്. കോഡിൽ ഒരു ചെറിയ ബ്രാക്കറ്റ് മാറിയാൽ പോലും ഇൻഫ്രാസ്ട്രക്ചർ തകരാൻ അത് കാരണമാകും.

ടെറാഫോം ഫയലുകളുടെ നിയമങ്ങൾ ഒരു **"ഭക്ഷണശാലയുടെ മെനു"** സെറ്റ് ചെയ്യുന്നതുപോലെ ലളിതമായി താഴെ വിവരിക്കുന്നു.

---

### **1. ബ്ലോക്ക് സ്ട്രക്ചർ (The Container Rule)**

ടെറാഫോമിലെ ഓരോ കാര്യവും ഒരു **Block** ഉള്ളിലാണ് എഴുതുന്നത്. ഇത് വസ്ത്രങ്ങൾ അലമാരയിൽ അടുക്കി വെക്കുന്നതുപോലെയാണ്. ഓരോ സെക്ഷനും കൃത്യമായ അതിർത്തിയുണ്ടാകും.

* **നിയമം:** എല്ലാ ബ്ലോക്കുകളും `{` (Open bracket) കൊണ്ട് തുടങ്ങുകയും `}` (Close bracket) കൊണ്ട് അവസാനിക്കുകയും വേണം.
* **റിയൽ വേൾഡ് എക്സാമ്പിൾ:** ഒരു റെസ്റ്റോറന്റിലെ 'പാനീയങ്ങൾ' എന്ന സെക്ഷൻ തുടങ്ങുമ്പോൾ മുതൽ അവസാനിക്കുന്നത് വരെ ആ സെക്ഷനിലെ ഐറ്റങ്ങൾ മാത്രമേ അതിൽ കാണാൻ പാടുള്ളൂ.

```hcl
resource "aws_instance" "food_app" {  # ബ്ലോക്ക് തുടങ്ങുന്നു
  ami           = "ami-12345"         # ഉള്ളിലെ ഡീറ്റെയിൽസ്
  instance_type = "t2.micro"
}                                     # ബ്ലോക്ക് അവസാനിക്കുന്നു

```

### **2. ഇൻഡന്റേഷൻ (The Two-Space Rule)**

ഓരോ ബ്ലോക്കിനുള്ളിലും എഴുതുന്ന വരികൾ അല്പം ഇടം വിട്ടായിരിക്കണം തുടങ്ങേണ്ടത്. സാധാരണയായി **രണ്ട് സ്പേസ് (2 spaces)** ആണ് ടെറാഫോമിൽ ഇതിനായി ഉപയോഗിക്കുന്നത്.

* **എന്തിന്:** ഏത് ഡീറ്റെയിൽ ഏത് ബ്ലോക്കിന്റെ ഉള്ളിലാണ് എന്ന് പെട്ടെന്ന് തിരിച്ചറിയാൻ വേണ്ടിയാണിത്.
* **റിയൽ വേൾഡ് എക്സാമ്പിൾ:** ഒരു മെനു കാർഡിൽ 'ജ്യൂസുകൾ' എന്ന ഹെഡിംഗിന് താഴെയായി ആപ്പിൾ, ഓറഞ്ച് എന്നിങ്ങനെ അല്പം ഉള്ളിലേക്ക് നീക്കി എഴുതുന്നതുപോലെയാണിത്.

```hcl
resource "aws_security_group" "web_traffic" {
  # ഈ ഇൻഡന്റേഷൻ (2 spaces) ശ്രദ്ധിക്കുക
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

```

### **3. അലൈൻമെന്റ് (The Equal-to Sign Rule)**

പ്രൊഫഷണൽ എൻജിനീയർമാർ കോഡ് എഴുതുമ്പോൾ എല്ലാ `=` ചിഹ്നങ്ങളും ഒരേ വരിയിൽ വരാൻ ശ്രമിക്കാറുണ്ട്. ഇത് വായിക്കാൻ വളരെ സുഖമുള്ളതാണ്.

* **നിയമം:** `key = value` രീതിയിൽ എഴുതുമ്പോൾ എല്ലാ വരികളിലെയും `=` ഒരേ ലെവലിൽ വരണം.
* **പ്രധാന കമാൻഡ്:** നിങ്ങളുടെ കോഡ് തനിയെ ഭംഗിയായി ഇരിക്കാൻ ഈ കമാൻഡ് അടിക്കാം:
```bash
terraform fmt

```


*(ഇത് ചെയ്താൽ ടെറാഫോം നിങ്ങളുടെ കോഡിലെ ഇൻഡന്റേഷൻ തനിയെ ശരിയാക്കും).*

---

### **4. നെയ്മിംഗ് കൺവെൻഷൻ (Naming Rule)**

നിങ്ങളുടെ ഫയലുകളുടെയും റിസോഴ്‌സുകളുടെയും പേരുകൾ നൽകുമ്പോൾ ശ്രദ്ധിക്കേണ്ട ചില കാര്യങ്ങൾ:

* പേര് ലളിതമായിരിക്കണം.
* സ്പേസ് ഉപയോഗിക്കരുത്, പകരം **അണ്ടർസ്കോർ (_)** ഉപയോഗിക്കുക (ഉദാ: `food_delivery_server`).
* **ഉദാഹരണം:** "റെസ്റ്റോറന്റ് 1" എന്ന് പേരിടുന്നതിനേക്കാൾ "Kerala_Food_Corner" എന്ന് പേരിടുന്നതാണ് ഒരു ബിസിനസ്സിൽ നല്ലത്. അത് ആ റിസോഴ്‌സിനെ വേഗം തിരിച്ചറിയാൻ സഹായിക്കും.

---

### **5. കമന്റിംഗ് (The Communication Rule)**

ഭാവിയിൽ വേറൊരു എൻജിനീയർ നിങ്ങളുടെ കോഡ് നോക്കുമ്പോൾ അത് എന്താണെന്ന് മനസ്സിലാക്കാൻ കമന്റുകൾ ചേർക്കണം.

* **# (Hash):** ഒറ്റ വരി കമന്റിന് ഉപയോഗിക്കുന്നു.
* **/* ... */:** ഒന്നിലധികം വരികൾക്ക് ഉപയോഗിക്കുന്നു.

**റിയൽ വേൾഡ് പ്രൊഡക്ഷൻ എക്സാമ്പിൾ:** "ഈ പോർട്ട് 8000 തുറക്കുന്നത് നമ്മുടെ ഫുഡ് ഡെലിവറി ആപ്പിന്റെ API-ക്ക് വേണ്ടിയാണ്" എന്ന് കോഡിന്റെ വശത്ത് എഴുതി വെക്കുന്നത് ടീമിലെ മറ്റുള്ളവർക്ക് ഉപകരിക്കും.

---

### **അടുത്ത നടപടി:**

നമ്മൾ ഇപ്പോൾ നിർമ്മിച്ച `main.tf` ഫയലിലെ ഇൻഡന്റേഷൻ ശരിയാക്കാൻ താഴെ പറയുന്ന കമാൻഡ് ഒന്ന് റൺ ചെയ്തു നോക്കാമോ?

```bash
terraform fmt

```

ഇതിന് ശേഷം ഫയൽ നോക്കുമ്പോൾ കോഡ് കൂടുതൽ വൃത്തിയായി ഇരിക്കുന്നത് നിങ്ങൾക്ക് കാണാം.
